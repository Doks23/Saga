# Saga — Developer Handover Document

## Purpose

Saga is a browser-based AI storytelling tool that converts book titles or original scripts into audio-visual content. Users enter a book name, get an AI-generated summary script, convert it to speech, generate matching visuals (thumbnail, concept art, book cover), and optionally produce a video with all assets composited on a canvas.

The project was started as an experimental prototype — likely via Google AI Studio's app export — and is not yet production-ready.

```

│  ┌────────────┐   ┌──────────────┐   ┌─────────────┐              │
│  │  App State  │──►│  Gemini API  │──►│  Output     │              │
│  │  (React)    │   │  @google/    │   │  - Script   │              │
│  │             │   │  genai       │   │  - Audio    │              │
│  │             │   │              │   │  - Images   │              │
│  │             │   │              │   │  - Video    │              │
│  └────────────┘   └──────────────┘   └─────────────┘              │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
         │
         ▼
  Google Gemini API
  (cloud)
```

- **Frontend only** — No backend server. The browser calls the Gemini API directly.
- **Single page** — No routing; a single `<App />` component manages everything.
- **State** — All application state is in React `useState` hooks. No external state management.
- **Build** — Vite compiles TypeScript + JSX into bundled files.

## Main Execution Flow

### Saga Mode Flow (Book → Script → Audio → Visuals → Video)

```
User enters book name + author
        │
        ▼
[Find Book] ──► Gemini: identify exact title/author ──► Confirmation card
        │
        ▼
[Generate Script] ──► Gemini: create script + YouTube metadata (JSON)
        │
        ├──► Script displayed in editable textarea
        ├──► Metadata shown in card (title, description, hashtags)
        └──► Auto-triggers: generateVisuals()
                │
                ▼
        Gemini (flash-image) generates 3 images:
          - YouTube Thumbnail
          - Concept Art (16:9 background)
          - Book Cover (vertical)
                │
                ▼
[Generate Audio] ──► Gemini TTS: splits text into 2500-char chunks
        │              streams PCM16 audio per chunk
        ▼
    Audio assembled into WAV blob → sticky bottom player
        │
        ▼
[Open Video Studio] ──► Canvas compositing: bg image + cover overlay + text
        │
        ▼
[Render Video] ──► MediaRecorder: captures canvas + audio into WebM
```

### Story Studio Flow (Script → Audio → Visuals)

```
User pastes script with scene/SFX notation
        │
        ▼
[Generate Audio Story] ──►
  1. generateMetadata(script) → title, description, hashtags
  2. generateVisuals(title, script) → thumbnail + concept art
  3. generateAudio(script, isScriptMode=true) → cleans script, TTS audio
```

### Internal Functions

| Function | Purpose |
|---|---|
| `getFileName()` | Generates sanitized download filenames (Saga_Title_Type_DDMM) |
| `cleanJsonText()` | Strips markdown code fences and extracts JSON object from LLM output |
| `safeJsonParse()` | Parses JSON with fallback sanitization for unescaped control characters |
| `splitText()` | Splits text into ~2500-char chunks at paragraph/sentence boundaries for TTS |
| `decode()` | Base64 → Uint8Array |
| `pcm16ToWavBlob()` | Converts raw PCM16 data to WAV file with proper RIFF/WAVE headers |
| `writeString()` | Writes ASCII string into DataView (for WAV header construction) |

## Data Model / Key Entities

The app has no backend persistence. All data is ephemeral React state:

```
Book { title: string, author: string }
Script { text: string, wordCount: number }
YouTubeMetadata { title, description, hashtags, tagline }
Audio { url: string (blob), format: "wav" }
Visuals {
  thumbnail: string (data URL),
  concept: string (data URL),
  cover: string (data URL)  // Saga mode only

Video { blob: WebM, url: string }
```

## Configuration Details

### Environment Variables

Set via `.env.local`:

```
GEMINI_API_KEY=<your_key>
```

Injected at build time by `vite.config.ts` into both `process.env.API_KEY` and `process.env.GEMINI_API_KEY` (duplicated).

### Hardcoded Gemini Models

| Purpose | Model ID |
|---|---|
| Script/Metadata generation | `gemini-2.5-flash` |
| Image generation | `gemini-2.5-flash-image` |
| Text-to-Speech | `gemini-2.5-flash-preview-tts` |

### Voices

Five prebuilt voices: `Puck`, `Charon`, `Kore`, `Fenrir`, `Zephyr`

### Audio Parameters

- Sample rate: 24000 Hz
- Format: PCM16 → WAV
- Chunk size: ~2500 characters
- Playback speed: 0.8x – 1.2x

## Deployment Details

There is no deployment configuration. The project currently runs only locally.

To prepare for deployment:
1. Run `npm run build` to produce the `dist/` folder
2. Deploy `dist/` to any static host (Vercel, Netlify, Cloudflare Pages, etc.)

**Important:** Because the GEMINI_API_KEY is injected at build time into the client bundle, it will be visible in the browser's dev tools. A backend proxy should be added before any public deployment.

## Risks and Weak Areas

1. **API key exposure** — `process.env.API_KEY` is inlined into the client bundle at build time. Anyone viewing the source can extract it.
2. **No tests** — Zero coverage. Regressions cannot be caught automatically.
3. **Monolithic file** — `index.tsx` at 1715 lines is difficult to maintain, review, and debug.
4. **No error boundaries** — A JavaScript error in any component unmounts the entire app.
5. **Hardcoded model names** — If Gemini deprecates a model (e.g., `gemini-2.5-flash-preview-tts`), the app breaks until code is updated.
6. **Importmap conflicts** — `index.html` defines an importmap pointing to CDN versions of React etc., while Vite bundles these same modules. This can cause React to load twice with separate contexts, leading to hooks errors.
7. **No lock file** — `package-lock.json` is not generated/committed, so installs are not deterministic.
8. **No accessibility** — Buttons lack ARIA labels, keyboard navigation is partial, no focus management for modals.
9. **No mobile responsiveness** — The app is usable on mobile but not optimized.
10. **Duplicate `<script>` tags** — `index.html:27` duplicates `index.tsx` module load (line 26 also loads it), causing double execution.

## Suggested Improvement Roadmap

1. **Immediate:**
   - Delete duplicate `<script>` tag in `index.html`
   - Generate and commit `package-lock.json`
   - Create `.env.example`
   - Add ESLint + Prettier config

2. **Short-term:**
   - Split `index.tsx` into components (`App.tsx`, `VideoModal.tsx`, `Logo.tsx`, `utils.ts`, `api.ts`)
   - Split `index.css` into modular CSS files
   - Add error boundaries for script/audio/visuals sections
   - Add a lightweight API proxy (Vite server middleware or simple Express/Cloudflare Worker)

3. **Medium-term:**
   - Add Vitest for unit tests (start with `safeJsonParse`, `splitText`, `pcm16ToWavBlob`)
   - Add accessible loading skeletons
   - Implement keyboard-navigable modals (trap focus, Escape to close)
   - Add responsive mobile layout

4. **Long-term:**
   - Consider a backend (Node.js/Next.js API routes) to protect the API key
   - Add user authentication and saved projects
   - Add database persistence (SQLite, Supabase, or similar)
   - Add media asset management (storage for generated images/audio)
